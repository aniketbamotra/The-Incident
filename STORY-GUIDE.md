# The Incident — Story Author's Guide

How the game actually works, and what you need to write. Everything here
reflects the built app, not aspirations — so anything you author maps directly
onto a screen or a mechanic.

---

## The shape of a game

A group of players, one evening, one physical room they keep gathering in and
dispersing from. **16 is the classic size, but you set the player count when you
create the game (8–24)** — everything below scales to whatever you choose. You
(the host) author everything *before* the night: one character per seat, the
clues that drop during play, and the phase pacing. During the night you drive a
dashboard — releasing clues, opening votes, calling everyone back together — and
it ends with a projected finale that exposes every lie.

The drama doesn't come from solving a fixed mystery. It comes from **what each
player chooses to reveal, hide, or lie about** when they're handed private
information that contradicts someone else's.

---

## Fastest path: generate it with AI and import

You don't have to type 16 character sheets by hand. The setup screen has an
**Import** button (top-right) that loads a single JSON file describing every
character and clue at once. So the workflow is:

1. Hand an AI this guide (or just the schema below) and ask it to write the
   story as JSON — "16 characters with secrets, objectives, a connected
   find-graph, and 6–8 paired clues across phases 1–5, matching this schema."
2. Create a game in the app, click **Import**, paste or upload the JSON.
3. Review and tweak in the Characters/Clues tabs, then Launch.

Importing **replaces** the current setup (it clears existing clues and resets
all seats), so it's meant as a one-shot load you then fine-tune. Download a
working example from the **"Download the template"** link in the import dialog
(it's also at `/public/story-template.json`).

### Import JSON schema

```json
{
  "game": { "name": "Optional — sets the game name" },
  "characters": [
    {
      "name": "Vera",                 // required, unique; becomes the character name
      "role": "The Fixer",
      "public_persona": "...",
      "secret": "...",
      "objective": "...",
      "find": "Cass",                  // optional; must match another character's name
      "find_question": "...",
      "hide_description": "...",        // host-only note
      "memories": ["...", "...", "..."] // up to 3 (extras ignored)
    }
    // up to your game's seat count (default 16; 8–24)
  ],
  "clues": [
    {
      "phase": 1,                      // 1–5
      "public_announcement": "...",    // required — everyone sees this on release
      "master_content": "...",         // required — the truth, finale only
      "implicating": {
        "character": "Vera",           // required; must match a character name
        "private_content": "...",      // required
        "implicates_self": true        // optional, default false
      },
      "neutral": {
        "character": "Cass",           // required; must differ from implicating
        "private_content": "..."       // required
      }
    }
  ],
  "phases": [ { "phase": 1, "notes": "..." } ]  // optional → phase planning notes
}
```

The importer **validates before changing anything** — if a clue points at a
character that doesn't exist, a name is duplicated, there are more than 16
characters, or a required field is missing, it lists the problems and imports
nothing. Fix the file and try again. (Notes from `phases` are saved into your
local planning notes on the Phases tab.)

> The rest of this guide explains each field and mechanic in depth — useful
> context to give the AI, and to understand what you're tuning after import.

## What you author

There are exactly three things to write: **characters**, **clues**, and
**phase notes**. Everything else (trust, tips, votes, the social graph) is
generated by players during play.

### 1. Characters (×16)

Each of the 16 seats is one character. Fields, and where each one surfaces:

| Field | Who sees it | Where it appears |
|---|---|---|
| **Character name** | everyone | name tag, story graph, trust list, reveal |
| **Role** | everyone | shown under the name (e.g. "The Fixer") |
| **Public persona** | everyone | their "You" tab, how they present |
| **Secret** | only them | red box on their "You" tab; exposed at finale |
| **Objective** | only them | green box on their "You" tab; scored at finale |
| **Finds (target)** | only them | "Who to find" card — a directed person to seek |
| **Question to ask them** | only them | the pretext for approaching their target |
| **What to hide** | **host only** | a private planning note (not shown to the player) |
| **Memory 1 / 2 / 3** | only them | first-person fragments that *decay over time* |

Write the three **memories as first-person fragments** — "I remember the glass
breaking," not "Vera heard glass." They display in italics and fade as they
decay (see Memory decay below).

> **The "Finds" graph matters.** As you assign each character's target, the
> setup screen draws a live web with arrows. Anyone nobody is looking for is
> flagged **amber ("unsought")**. Aim for a fully connected graph — ideally
> cycles, so every person is both seeking and sought. Isolated characters have
> no social pull and drift to the edges of the room.

### 2. Clues

A clue is the engine of the game. **Every clue goes to exactly two players** —
one gets the "implicating" version, the other the "neutral" version — plus a
public line everyone sees and a private truth only revealed at the finale.

| Part | Who sees it | When |
|---|---|---|
| **Public announcement** | everyone | the moment you release the clue |
| **Master content (truth)** | nobody until finale | finale Phase 1 |
| **Player A — implicating version** | only Player A | when released |
| ↳ *"implicates the recipient" toggle* | — | flags whether it points at A |
| **Player B — neutral version** | only Player B | when released |

When you release a clue, the public announcement posts to **everyone's Intel
tab**, and the two assigned players each get a private "you have a clue" prompt.
Each of them then privately decides: **Reveal** or **Sit on it**. What happens
to the group depends on both choices (see Clue decisions below).

Spread clues across **phases 1–5**. A good clue creates a *contradiction*: A's
implicating version and B's neutral version describe the same moment
differently, so if one reveals and the other's account doesn't match, hiding
looks deliberate.

### 3. Phase notes

Five phases pace the night (e.g. Arrival → Dispersal → The turn → Regather →
Reckoning). The **Phases tab is your private planning space** — gathering
prompts, vote prompts, reminders. These notes live only on your device; the
*live* phase control (advance, release, open vote) is on the dashboard.

## The mechanics (how play behaves)

### Memory decay

While a game is **active**, a background job runs every ~15 minutes and drops
each memory's strength by a random **8–12 points**. As a memory fades:

- the text literally dims (opacity tracks strength),
- below **40%** it shows *"You're no longer certain about this."*

Design memories that **matter early and erode**: a precise detail someone is
sure of at 9pm becomes a hazy maybe by 11pm. That uncertainty is deliberate
fuel for doubt and contradiction.

### Clue decisions (the heart of it)

Each of the two recipients chooses **Reveal** (publishes their version to the
whole group, attributed to them) or **Sit on it** (withheld, but recorded). The
group outcome:

| Player A | Player B | What the group gets |
|---|---|---|
| Reveal | Reveal | **Both** versions post → the contradiction is visible to all |
| Reveal | Hide | Only the **revealer's** version posts |
| Hide | Reveal | Only the **revealer's** version posts |
| Hide | Hide | The **neutral** version auto-posts, **no attribution** |

Key tension you're designing for: **the group always knows a clue dropped**
(the public announcement went out), and each recipient knows *someone else also
got a version*. So hiding is never invisible — if your partner reveals and your
silence contradicts them, it reads as a choice.

### Trust

Every player rates every other player **0–100** on a slider. **Ratings are
public** — everyone sees them. Your own "trust score" is the average of what
others rated you. This is social pressure made literal.

### Votes

When you (host) **open a vote** for the current phase, every player gets a
full-screen "Who do you suspect most?" prompt. They pick one suspect and can
change it until you close the vote. You see **live aggregate counts**; individual
votes stay anonymous until the finale.

### Tips

Players send **anonymous** tips — to everyone or to one specific person. The
sender is never shown, to anyone, ever (until... no — never; it's not even in
the finale). Pure rumor injection.

### Gathering / dispersal

Hitting **"Call gathering"** flips every player's screen to a pulsing banner:
*"The group is gathering. Go to the main room."* Use it to pull the room back
together for a clue drop or a vote.

### Social gravity (silent)

Every time a player tags someone in a story event or rates their trust, the
system silently records an **interaction** between them. Players never see this
during play — it only surfaces as the finale's heat map, showing who actually
orbited whom.

## The finale (host projects this on a TV)

Five phases, you click through them:

1. **What actually happened** — the master truth of every clue + a timeline
   reconstructed from the story graph.
2. **Every lie** — who got what, who revealed, who hid; contradictions
   highlighted; stat counts.
3. **The trust map** — a force-directed graph. Toggle **Public** (what people
   claimed) vs **Real** (actual honesty, derived from clue choices). Node size =
   how much others trusted them.
4. **Epilogues** — one card per player: conversations, truths vs lies, whether
   their objective was met, their secret (was it discovered?), who they trusted
   most and how honest that person really was.
5. **Leaderboard** — final ranking, revealed bottom-up.

> The finale is **meant to be projected by the host**. Players watching on their
> own phones see a reduced version (the app hides other players' private clue
> data from them) — so run it on the big screen.

### How scoring works

Because the schema doesn't store a "culprit" or an objective checkbox, scores
are derived from **observable behavior**:

| Component | Points |
|---|---|
| Objective met (revealed every clue that implicated you) | +300 |
| Each correct vote (voted for the most-implicated player) | +50 |
| Each clue revealed honestly | +100 |
| Trust accuracy (how well your ratings matched real honesty) | up to +150 |
| Each clue hidden (a withholding) | −75 |

This rewards honesty and accurate read of others, and punishes withholding — so
a player who lies their way through can still "win socially" but rarely tops the
board.

## Authoring checklist

Write these, in this order:

- [ ] **One character sheet per seat** (your chosen player count, default 16) —
      name, role, public persona, secret, objective, 3 memory fragments.
- [ ] **The find-graph** — assign each character's target + question so the
      setup web shows *no amber (unsought) nodes*. Cycles are good.
- [ ] **Clue set** — for each clue: the two players, which is implicating
      (and whether it implicates them), each private version, the public line,
      and the master truth. Spread across phases 1–5. Aim for several
      contradiction pairs.
- [ ] **Phase notes** — a one-line gathering prompt and vote prompt per phase.

## Design tips for maximum drama

- **Tie secrets to clues.** A character's private clue should brush against
  their secret or objective, so revealing it costs them something.
- **Build contradiction pairs.** The same event seen two ways means someone's
  silence will eventually look like a lie.
- **Make objectives collide.** Two characters whose objectives require the same
  thing (the same person's trust, the same hidden fact staying buried) generate
  natural conflict.
- **Front-load certainty, let it rot.** Put a sharp, checkable detail in a
  memory; by late game its decay makes the holder doubt themselves on the stand.
- **Seed a few asymmetric finds.** A character told to find someone who has no
  reason to talk to them creates pursuit and avoidance — both tracked silently
  for the finale.
- **Don't strand anyone.** An unsought character with a weak objective just
  stands in the corner. Give every seat a reason to be approached.
